# Persistence — Spatial Schema & Migrations

## Rules

- **All schema changes go through Alembic.** Never alter a table by hand in any environment.
- **Never delete a migration file** once it has been applied anywhere. Write a new migration to undo a change.
- Migration filenames: `NNNN_short_description.py` with a 4-digit prefix incremented by 1.

## Commands

```bash
# From backend/
uv run alembic upgrade head              # apply all pending migrations
uv run alembic current                   # show current revision
uv run alembic downgrade -1              # revert one migration

# Create a new migration
uv run alembic revision --autogenerate -m "describe_change"   # from model diff
uv run alembic revision -m "describe_change"                   # blank (manual)
```

`DATABASE_URL` must be set and the database must be running before any alembic command.

## asyncpg → psycopg2 URL conversion

`DATABASE_URL` uses `postgresql+asyncpg://` for the app. Alembic uses a synchronous connection internally. `env.py` converts the URL automatically — you don't need to change anything. `psycopg2-binary` must be installed (it's in `pyproject.toml`).

## CRS conventions

| SRID | Name | Used for |
|------|------|----------|
| 4326 | WGS84 (lat/lon degrees) | All stored geometry columns |
| 3857 | Web Mercator (metres) | Buffer calculations only — transform in, buffer, transform back to 4326 |

Never store geometry in 3857. Never compute distances or areas directly in 4326 (degrees are not metres).

## Spatial schema

### `land_cover_polygons`
Polygon geometries for ESA WorldCover data. Queried via `ST_Intersects` against the analysis area.

### `deforestation_alerts`
Point geometries for GFW Integrated Alerts.

**De-normalised coordinates:** `longitude` and `latitude` Float columns duplicate the geometry. This exists because PostGIS geometry columns cannot participate in `UNIQUE` constraints. The `UNIQUE(longitude, latitude, alert_date, source)` constraint is what enables idempotent re-ingestion (`ON CONFLICT DO NOTHING`). Do not remove these columns.

**Spatial index:** alerts are queried via `ST_Within(geometry, analysis_area)`. If this becomes slow as alert volume grows, add a GIST index on `deforestation_alerts.geometry`.

## When to autogenerate vs write manually

**Autogenerate is fine for:**
- Adding plain columns (VARCHAR, INTEGER, FLOAT, DATE)
- Adding or dropping indexes on non-geometry columns

**Write manually for:**
- PostGIS geometry columns — GeoAlchemy2 types may not autogenerate correctly
- Spatial indexes: `CREATE INDEX ... USING GIST (geometry)`
- `UNIQUE` constraints that involve de-normalised coordinate columns
- Any raw DDL (`op.execute(...)`)

**Always review autogenerated files** before applying — autogenerate can produce incorrect diffs when geometry columns or custom types are involved.

## Spatial index example (manual migration)

```python
def upgrade() -> None:
    op.execute(
        "CREATE INDEX ix_deforestation_alerts_geometry "
        "ON deforestation_alerts USING GIST (geometry)"
    )

def downgrade() -> None:
    op.execute("DROP INDEX IF EXISTS ix_deforestation_alerts_geometry")
```
