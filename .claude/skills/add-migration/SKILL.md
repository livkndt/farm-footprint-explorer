You are adding a new Alembic migration to Farm Footprint Explorer. The change needed is: $ARGUMENTS

Follow this workflow in order.

## 1. Decide: autogenerate or write manually?

**Write manually** if the change involves any of:
- A PostGIS geometry column (GeoAlchemy2 types do not autogenerate correctly)
- A spatial index: `CREATE INDEX ... USING GIST`
- A `UNIQUE` constraint on de-normalised coordinate columns
- Raw DDL with no SQLAlchemy ORM equivalent

**Autogenerate is fine** for:
- Adding plain columns (VARCHAR, INTEGER, FLOAT, DATE, BOOLEAN)
- Adding or dropping non-spatial indexes
- Adding new plain tables with no geometry columns

## 2. Update the ORM model (if adding a column or table)

Edit `backend/app/models/footprint.py` first. Alembic autogenerate diffs the ORM models against the live schema, so the model must reflect the intended end state before generating.

CRS reminder:
- Geometry columns: always `Geometry("POINT", srid=4326)` or `Geometry("POLYGON", srid=4326)` — never 3857 in stored columns
- De-normalised coordinates: if adding a geometry column that needs a `UNIQUE` constraint, also add `Float` columns for the coordinates (see `DeforestationAlert` as the pattern)

## 3. Create the migration file

**Autogenerate:**
```bash
cd backend
uv run alembic revision --autogenerate -m "short_description_of_change"
```

**Manual:**
```bash
cd backend
uv run alembic revision -m "short_description_of_change"
```

Name the file with the next 4-digit prefix (check `alembic/versions/` for the current highest number).

## 4. Review the generated file — always, even for autogenerate

Open the new file in `backend/alembic/versions/`. Check:
- `down_revision` points to the correct previous migration
- `upgrade()` contains exactly the intended changes and nothing else
- `downgrade()` correctly reverses the upgrade
- Geometry columns use `op.execute()` with raw DDL, not autogenerated type mappings
- Spatial indexes use `USING GIST`, not the default B-tree

If autogenerate produced incorrect geometry column DDL, replace it with `op.execute()`:

```python
def upgrade() -> None:
    op.execute(
        "ALTER TABLE deforestation_alerts "
        "ADD COLUMN new_geom geometry(POINT, 4326)"
    )

def downgrade() -> None:
    op.execute(
        "ALTER TABLE deforestation_alerts DROP COLUMN new_geom"
    )
```

Spatial index example:
```python
def upgrade() -> None:
    op.execute(
        "CREATE INDEX ix_deforestation_alerts_new_geom "
        "ON deforestation_alerts USING GIST (new_geom)"
    )

def downgrade() -> None:
    op.execute("DROP INDEX IF EXISTS ix_deforestation_alerts_new_geom")
```

## 5. Apply the migration

The database must be running (`docker compose up -d db`).

```bash
cd backend
DATABASE_URL=postgresql+asyncpg://farmuser:farmpass@localhost:5432/farmfootprint \
uv run alembic upgrade head
```

Verify it applied:
```bash
DATABASE_URL=postgresql+asyncpg://farmuser:farmpass@localhost:5432/farmfootprint \
uv run alembic current
# Should show your new revision as (head)
```

Spot-check the schema in psql:
```bash
docker compose exec db psql -U farmuser -d farmfootprint -c "\d table_name"
```

## 6. Test the downgrade (optional but good practice for destructive changes)

```bash
DATABASE_URL=postgresql+asyncpg://farmuser:farmpass@localhost:5432/farmfootprint \
uv run alembic downgrade -1

# Re-apply
uv run alembic upgrade head
```

## 7. Run the test suite

```bash
DATABASE_URL=postgresql+asyncpg://farmuser:farmpass@localhost:5432/farmfootprint \
TEST_DATABASE_URL=postgresql+asyncpg://farmuser:farmpass@localhost:5432/farmfootprint \
uv run pytest -v
```

## 8. Commit

Commit the migration file and any ORM model changes together. Never commit a model change without the migration or vice versa.
